# 13 企业实践与结课项目：构建通用型大模型知识图谱系统

## 引言

最后一课把所有内容组合成一个企业项目：构建一个通用型 LLM 知识图谱系统，并支持 GraphRAG 和 Agent 查询。

## Capstone 任务

选择一个资料集合，例如：

- 公司产品文档。
- 学术论文集合。
- 客服工单。
- 开源项目 README 和源码说明。
- 业务合同与政策文档。

构建：

1. 数据源登记。
2. 文档读取与 chunk。
3. LLM 抽取实体和关系。
4. Neo4j 写入。
5. `HAS_ENTITY` 溯源。
6. 向量索引与全文索引。
7. KNN 相似 chunk。
8. 重复实体合并。
9. 孤立节点检查。
10. 社区检测与社区摘要。
11. 多模式 GraphRAG 问答。
12. Agent 查询工具。

## 案例驱动任务书

建议使用同一个贯穿案例：把一个开源项目或产品文档构建成知识图谱。

最终交付需要包含：

- 一份术语表：解释实体、关系、路径、多跳、社区、向量索引。
- 一份 schema：列出节点类型、关系类型、主键和方向。
- 一条样例路径：例如 `功能 -> 文件 -> 服务 -> 客户问题`。
- 三个 GraphRAG 问题：局部事实、多跳关系、全局总结。
- 一次工程变更：模拟文档更新、删除或失败续跑。
- 一份评估报告：实体准确率、关系准确率、答案引用率、延迟和成本。

## 企业上线清单

- 数据权限：谁能读哪些文档和实体？
- 写入权限：是否允许自动写图谱？
- Schema 版本：新增 label 和 relationship 如何审批？
- 评估：实体召回、关系准确、答案引用、延迟、成本。
- 可观测性：token usage、处理时间、失败原因、重试次数。
- 数据生命周期：文档删除后实体和社区如何清理？
- 人工审核：高风险关系是否需要确认？

## 从本项目借鉴的最佳实践

`llm-graph-builder` 值得借鉴的地方包括：

- 多数据源输入。
- Document 状态机和重试。
- Chunk 与实体分层。
- LLM 抽取 schema 约束。
- Neo4j 向量和全文索引。
- 后处理任务入口。
- 多 chat mode 对比。
- 社区摘要支持全局问题。

## 最终作品要求

你的最终系统至少能回答三类问题：

- 局部事实：某个实体是什么，来自哪里？
- 多跳关系：两个实体如何关联？
- 全局总结：这批资料主要有哪些主题和社区？

如果能做到“答案引用原文 chunk，并展示相关实体路径”，就已经具备企业级原型价值。

## 小结

知识图谱不是一项单点技术，而是一套把数据、模型、图数据库、检索、问答、治理连接起来的系统工程。LLM 让它更容易启动，Agent 让它更有应用想象力，而工程治理决定它能否长期可靠运行。
