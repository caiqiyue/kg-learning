# 01 知识图谱的起源：从字符串到事物

## 引言

知识图谱最容易被误解成“把数据画成节点和边”。更准确地说，它是一种把**事物、概念、属性、关系和上下文**组织成可查询结构的方式。它关心的不是文本里出现了多少次某个词，而是“这个词指向哪个真实对象，它和其他对象有什么关系”。

Google 在 2012 年介绍 Knowledge Graph 时用了一个很有代表性的表达：搜索应该理解 things，而不只是 strings。比如用户搜索“Apple”，系统要判断它可能是公司、水果、唱片公司，甚至是某篇文档里的股票符号。知识图谱的核心价值，就是把模糊字符串落到明确实体上。

## 为什么需要知识图谱

传统数据库擅长保存表格化事实，传统搜索擅长找包含关键词的文本，向量检索擅长找语义相近的片段。但很多问题需要**关系理解**：

- “A 公司收购 B 公司后，哪些产品线可能受影响？”
- “这篇论文提到的方法和之前哪些算法有关？”
- “某个客户投诉背后涉及哪些订单、人员、系统和历史事件？”

这些问题不是单个片段能回答的。它们需要跨实体、跨关系、跨文档地追踪上下文。

## 知识图谱解决的问题

知识图谱把信息组织为实体和关系。例如：

```text
(Neo4j)-[:SUPPORTS]->(Cypher)
(llm-graph-builder)-[:USES]->(Neo4j)
(Chunk)-[:HAS_ENTITY]->(Entity)
```

这让系统可以回答：

- 某个实体来自哪些文档？
- 两个实体通过几跳关系相连？
- 哪些实体属于同一个社区？
- 检索到一个 chunk 后，能否扩展到相关实体和邻居关系？

## 和本项目的关系

本项目 `llm-graph-builder` 的 README 说明它能把 PDF、网页、YouTube 等非结构化数据转换成 Neo4j 知识图谱。它不是简单地存文本，而是把文档拆成 `Document`、`Chunk`、`Entity`、`Relationship`，再给 chunk 和实体建立索引、相似关系、社区摘要和问答入口。

这正是现代知识图谱的典型形态：不是孤立的数据库，而是服务检索、问答、可视化、治理和 Agent 行动的知识底座。

## 小结

知识图谱的起点是“让机器理解事物之间的关系”。它从搜索、语义网、数据库和 NLP 中发展而来，在 LLM 时代重新变得重要，因为大模型会生成语言，但企业系统还需要**可追踪、可更新、可校验、可查询**的知识结构。
